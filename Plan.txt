Product name:
“Планы?” / Fest&Rest

Product summary:
This is a mobile-first social event discovery and planning app.
It combines:
1) discovering events from venues
2) social coordination with friends
3) planning together even when place/time are not fully known yet

This is NOT just an event listing app.
This is NOT just a messenger.
The core value is bridging the gap between “I found something interesting” and “we actually made a plan and went together”.

Core loop:
discover event → mark interest or save → create a shared plan → discuss and vote inside the plan → finalize place/time → attend → later repeat with the same people

Main product idea:
Users browse event recommendations from venues.
From any event they can quickly create a shared plan with friends.
A plan has chat, proposals, and voting.
Plans can also be created from scratch without exact place/time.
Groups exist as persistent friend circles for repeated planning.
Past plans can be repeated with the same composition.

Platform:
Mobile-first app, likely React Native / Expo or similar mobile-oriented architecture.

Language:
Russian UI

Design direction:
Warm, soft, friendly, premium consumer app.
No corporate dashboards.
No visual clutter.
Rounded cards, soft shadows, airy spacing.
Mobile-native feel.
The existing visual direction can be summarized as “Soft Shell”.

==================================================
1. PRODUCT VISION
==================================================

The app solves a real coordination problem:
People often find events or want to go somewhere, but the actual planning breaks down because:
- nobody wants to commit immediately
- exact place is not always known
- exact time is not always known
- group chats become chaotic
- voting and coordination happen in scattered messengers

This product should make the process feel lightweight:
- discover event
- tap one button
- invite people
- coordinate inside one object
- finalize
- later reuse the same social graph

The product is built around 3 main entities:
1. Event = public event from a venue
2. Plan = social coordination object
3. Group = persistent collection of people for reuse

Chat is not a standalone product layer.
Chat lives inside plans and groups.

2. PRODUCT PRINCIPLES
==================================================

Use these as hard principles:

1. Event feed must show EVENTS, not generic venue cards.
2. Planning must work even when exact place/time are unknown.
3. “Going” is too hard of a commitment on the feed; use softer interest there.
4. Save/bookmark is private.
5. Social proof is important:
   - friends interested
   - friends already have a plan
6. The key CTA is not “buy ticket” or “go now”, but creating a shared plan.
7. A plan is the main coordination unit.
8. A plan should support:
   - participants
   - statuses
   - proposals
   - voting
   - chat
   - finalization
9. Groups are reusable containers of people.
10. Past plans should be easy to repeat with the same people.

3. MAIN USER FLOWS
==================================================

Formalize these flows in detail.

FLOW A — Discover event and create plan from it
1. User opens Home Feed
2. User sees event card from a venue
3. User can:
   - mark soft public interest
   - save privately
   - open event details
   - tap “Планы?” to create a plan
4. If user taps “Планы?”:
   - open create-plan flow
   - event is prefilled
   - event time/place are known by default
   - user selects friends or a group
   - optionally enables “Встретиться до мероприятия”
   - sends invitations
5. Invited users receive plan invite
6. Plan appears in “Мои планы”
7. Participants discuss and possibly vote on pre-meet details
8. Event itself remains the anchor

FLOW B — Create generic plan without exact place/time
Example: “Пойдем в кино в субботу вечером, место решим позже”
1. User taps global plus/create action
2. User selects activity intent:
   - Кино
   - Кофе
   - Бар
   - Прогулка
   - Ужин
   - Спорт
   - Выставка
   - Другое
3. User can leave place undefined
4. User can leave time undefined or semi-defined
5. User invites friends or a group
6. Participants propose options
7. Voting happens
8. Final decision is confirmed
9. Plan becomes finalized and later past

FLOW C — Proposal and voting flow
1. Inside a plan, user proposes place or time
2. Proposal appears:
   - in details section
   - optionally as system/proposal card in chat
3. Participants vote
4. Voting is multi-select with maximum 2 choices
5. Plan creator or admin can finalize chosen option
6. Finalized option becomes official plan data

FLOW D — Reuse past plan / same group
1. User opens past plan or group
2. User taps “Повторить” or “Создать новый план с этой группой”
3. Same people are preselected
4. Activity can be reused or edited
5. New plan starts faster

FLOW E — Group-based planning
1. User opens group
2. Sees participants, past plans, upcoming plans
3. Taps create plan
4. Group members are preselected
5. Plan runs as normal coordination object
